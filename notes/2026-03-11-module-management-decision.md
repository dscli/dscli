## 背景

### 问题描述
在开发新的 Shell 包时，遇到了模块管理问题：
1. `internal/shell` 目录下创建了独立的 `go.mod` 文件
2. 导致子模块版本 (`mvdan.cc/sh/v3 v3.13.0`) 与主模块版本 (`v3.12.0`) 不一致
3. 无法从主模块直接导入子模块
4. 测试和集成示例无法正常运行

### 技术约束
1. Go 1.26 项目，使用 Go Modules
2. 项目已有完整的 `go.mod` 文件
3. `internal/shell` 是项目内部包，不是独立库
4. 需要保持依赖版本一致性

## 决策过程

### 考虑的方案

#### 方案1：使用 Go Workspace
```bash
# 创建 go.work 文件
go work init .
go work use ./internal/shell
```

**优点**：
- 支持多模块同时开发
- 允许不同模块使用不同依赖版本
- 官方推荐的多模块管理方式

**缺点**：
- 增加配置复杂度
- 对于内部包来说过于复杂
- 需要所有开发者了解 workspace 概念

#### 方案2：移除子模块，统一管理
```bash
# 移除 internal/shell/go.mod
rm internal/shell/go.mod
# 更新主 go.mod 依赖版本
go mod edit -require=mvdan.cc/sh/v3@v3.13.0
go mod tidy
```

**优点**：
- 简化项目管理
- 保持依赖版本一致性
- 符合内部包的设计原则
- 减少配置复杂度

**缺点**：
- 如果将来需要独立发布，需要重新配置
- 所有依赖都在主模块中管理

#### 方案3：保持子模块，使用 replace 指令
```go.mod
replace gitcode.com/dscli/dscli/internal/shell => ./internal/shell
```

**优点**：
- 保持模块独立性
- 便于将来提取为独立库

**缺点**：
- 配置复杂
- 开发体验不佳
- 测试和构建需要特殊处理

## 决策结果

### 选择的方案：方案2 - 移除子模块，统一管理

### 决策理由
1. **项目性质**：`internal/shell` 是项目内部包，不是独立库
2. **简化管理**：单一 `go.mod` 文件更易于维护
3. **依赖一致性**：确保所有代码使用相同的依赖版本
4. **开发体验**：减少配置复杂度，提高开发效率
5. **团队协作**：减少团队成员的学习成本

### 实施步骤
1. 删除 `internal/shell/go.mod` 文件
2. 删除 `internal/shell/go.sum` 文件
3. 更新主 `go.mod` 中的 `mvdan.cc/sh/v3` 到 `v3.13.0`
4. 运行 `go mod tidy` 同步依赖
5. 更新所有导入路径使用完整模块路径

### 具体操作
```bash
# 1. 移除子模块文件
rm -f internal/shell/go.mod internal/shell/go.sum

# 2. 更新主模块依赖
cd internal/shell
go mod edit -require=mvdan.cc/sh/v3@v3.13.0
cd ../..

# 3. 同步依赖
go mod tidy

# 4. 验证导入
cat > test.go << 'EOF'
package main
import "gitcode.com/dscli/dscli/internal/shell"
func main() {
    // 测试导入
}
EOF
go run test.go
rm -f test.go
```

### 正面影响
1. **简化配置**：只有一个 `go.mod` 文件需要维护

3. **构建可靠**：避免版本冲突和构建问题

2. **提取困难**：如果将来需要独立发布，需要重构

2. **文档记录**：记录模块结构，便于将来重构

## 经验教训

### 学到的经验
1. **内部包设计**：内部包通常不应该有自己的 `go.mod`
2. **依赖管理**：保持依赖版本一致性很重要
3. **简化优先**：在满足需求的前提下，选择最简单的方案
4. **文档记录**：技术决策需要详细记录，便于追溯

### 最佳实践
1. **单一模块**：对于单一项目，使用单一 `go.mod`
2. **版本同步**：定期更新依赖，保持版本一致
3. **测试验证**：任何模块变更都需要充分测试
4. **团队沟通**：技术决策需要团队共识
