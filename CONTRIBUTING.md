## 贡献者公约

欢迎为 dscli 项目做出贡献！本公约将帮助您了解如何参与项目开发，并确保项目保持高质量和一致性。

## 🚨 重要原则：代码来源规范

### 1. 只接受人和Dscli的代码

**核心原则**：本项目只接受以下两种来源的代码：

1. **人类编写的代码**：由人类开发者直接编写的代码
2. **Dscli生成的代码**：使用 dscli 工具本身协助生成的代码

**不接受**：
- 其他AI工具生成的代码（如ChatGPT、Copilot等）
- 未经修改的第三方代码片段
- 从其他项目直接复制的代码

### 2. 为什么这样做？

1. **保持项目一致性**：确保所有代码都遵循相同的设计哲学和编码风格
2. **实践我们推广的工具**：Dscli项目应该首先使用Dscli来开发
3. **避免知识产权问题**：确保所有代码都有明确的来源和授权
4. **质量可控**：人类和Dscli生成的代码都经过严格的质量控制

### 3. 如何验证代码来源？

贡献者需要：
1. 在提交信息中说明代码来源（人类编写 / Dscli协助）
2. 提供必要的开发过程记录
3. 确保代码符合项目规范

## 项目理念

dscli 是一个命令行工具，为 DeepSeek AI 提供本地化集成。我们的目标是：

1. 提供简洁、高效的 AI 辅助编程体验
2. 保持代码质量和可维护性
3. 实践我们推广的工具和方法论
4. 遵循最小化原则，避免不必要的复杂性
人类开发者编写代码后，可以：
1. 使用 dscli 分析代码变更
2. 让 dscli 协助编写提交信息
## 开发规范

### 代码风格
- 使用 Go 语言标准编码风格（gofmt）
- 遵循 Go 语言最佳实践
- 函数和变量使用有意义的命名
- 添加适当的注释和文档

### 文档规范
- 遵循最小文档原则，只维护必要的文档
- 避免创建不必要的发布说明文档
- 版本信息通过 Git 标签和提交历史获取
- 代码即文档，保持代码清晰可读

### 提交规范
- 提交信息使用约定式提交格式
- 每次提交应聚焦于单一功能或修复
- **鼓励使用 dscli 自身协助编写提交信息**
- **必须在提交信息中注明代码来源**

## 提交规范

### 核心原则
我们鼓励使用 dscli 来协助编写提交信息，确保提交信息的规范性和一致性。

### 为什么这样做？
1. 保持项目一致性
2. 实践我们推广的工具
3. 确保提交信息的规范性
4. 展示 AI 辅助开发的实际应用

### 如何操作？
人类开发者编写代码后，可以：
1. 使用 dscli 分析代码变更
2. 让 dscli 协助编写提交信息
3. 人工审核并完善提交信息
4. **在提交信息中注明代码来源**
5. 执行提交

### 提交信息格式
使用约定式提交格式，**必须包含代码来源说明**：
- `feat:` 新功能
- `fix:` 错误修复
- `docs:` 文档更新
- `style:` 代码格式调整
- `refactor:` 代码重构
- `test:` 测试相关
- `chore:` 构建过程或辅助工具

- 推送到远程仓库

## 📋 代码来源详细规范

### 1. 可接受的代码来源

#### 1.1 人类编写的代码
- **定义**：由人类开发者直接思考、设计和编写的代码
- **要求**：
  - 必须由贡献者亲自编写
  - 可以借鉴现有代码的设计思路，但不能直接复制
  - 必须理解每一行代码的含义和作用

#### 1.2 Dscli生成的代码
- **定义**：使用 dscli 工具协助生成的代码
- **要求**：
  - 必须使用 dscli 的 chat 或 reasoner 功能
  - 生成后必须由人类进行审查和修改
  - 必须确保代码符合项目规范
  - 需要在提交信息中说明使用了Dscli

### 2. 不可接受的代码来源

#### 2.1 其他AI工具
- ChatGPT、Copilot、Claude等AI工具生成的代码
- 任何非Dscli的AI辅助工具

#### 2.2 第三方代码
- 从其他项目直接复制的代码
- 未经修改的开源代码片段
- Stack Overflow等网站的代码片段

#### 2.3 自动生成工具
- 代码生成器生成的代码
- 模板引擎自动生成的代码
- 未经人类审查的自动化工具输出

### 3. 混合来源的处理

#### 3.1 人类+Dscli混合
- **允许**：人类编写部分代码，Dscli协助完成其他部分
- **要求**：必须在提交信息中详细说明各部分来源

#### 3.2 Dscli生成+人类修改
- **允许**：Dscli生成代码后，人类进行修改和优化
- **要求**：修改必须显著，不能只是简单调整

### 4. 验证方法

#### 4.1 提交信息验证
- 必须包含代码来源说明
- 必须说明使用了哪些工具
- 必须描述开发过程

#### 4.2 代码风格验证
- 检查代码是否符合项目规范
- 检查是否有其他工具的痕迹
- 检查代码逻辑是否一致

#### 4.3 过程记录验证
- 鼓励保留开发过程的对话记录
- 可以提供Dscli的交互历史
- 有助于审查者理解开发过程

### 1. 测试文件结构
测试文件必须与源文件一一对应：
```go
// 源文件：tools.go
// 测试文件：tools_test.go
```

### 2. 测试函数命名
使用描述性测试函数名：
```go
func TestFunctionName(t *testing.T) {
    // 基本功能测试
}

func TestFunctionName_EdgeCase(t *testing.T) {
    // 边界条件测试
}

func TestFunctionName_ErrorHandling(t *testing.T) {
    // 错误处理测试
}
```

### 3. 测试用例组织
使用表格驱动测试：
```go
func TestFunctionName(t *testing.T) {
    testCases := []struct {
        name     string
        input    string
        expected string
        wantErr  bool
    }{
        // 正常路径
        {"normal case", "input", "expected", false},
        // 边界条件
        {"empty string", "", "", false},
        // 错误路径
        {"invalid input", "invalid", "", true},
    }
    
    for _, tc := range testCases {
        t.Run(tc.name, func(t *testing.T) {
            // 测试逻辑
        })
    }
}
```

### 4. 测试覆盖率要求
- 总体覆盖率不低于80%
- 关键功能覆盖率不低于90%
- 错误处理路径必须覆盖

### 5. 测试质量指标
| 指标 | 要求 | 检查方式 |
|------|------|----------|
| **测试覆盖率** | ≥80% | `go test -cover` |
| **边界测试** | 必须包含 | 代码审查 |
| **错误测试** | 必须包含 | 代码审查 |
| **测试可读性** | 清晰易懂 | 代码审查 |
| **测试独立性** | 不依赖外部 | 独立运行 |

## 发布流程

### 版本号规则
- 主版本号：重大变更，可能不向后兼容
- 次版本号：新功能，向后兼容
- 修订号：错误修复，向后兼容

### 发布步骤
1. 确保所有测试通过
2. 创建 Git 标签（如 v0.4）
3. 标签信息包含版本亮点
4. 不创建单独的发布说明文档
5. 版本信息通过 Git 标签和提交历史获取

### 文档约定
- 不创建不必要的 RELEASE_*.md 文件
- 版本变更通过提交历史追溯
- 保持代码库简洁，减少维护负担
