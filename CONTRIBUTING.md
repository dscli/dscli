## Contributor Covenant

Welcome to the dscli project! This covenant will help you understand how to participate in project development and ensure the project maintains high quality and consistency.

## 🚨 Important Principle: Code Source Standards

### 1. Only Human and Dscli Code Accepted

**Core principle**: This project only accepts code from the following two sources:

1. **Human-written code**: Code directly written by human developers
2. **Dscli-generated code**: Code generated with the assistance of the dscli tool itself

**Not accepted**:

- Code generated by other AI tools (e.g., ChatGPT, Copilot, etc.)
- Unmodified third-party code snippets
- Code directly copied from other projects

### 2. Why This Policy?

1. **Maintain project consistency**: Ensure all code follows the same design philosophy and coding style
2. **Dogfood our own tools**: The dscli project should be developed using dscli first
3. **Avoid IP issues**: Ensure all code has a clear source and license
4. **Quality control**: Both human and Dscli-generated code undergo rigorous quality checks

### 3. How to Verify Code Source?

Contributors should:

1. Indicate the code source in commit messages (Human-written / Dscli-assisted)
2. Provide development process records when necessary
3. Ensure code conforms to project standards

## Commit Message Guide

### Core Principles

We encourage using dscli to help write commit messages, ensuring consistency and standards compliance.

### Why This Approach?

1. Maintain project consistency
2. Dogfood our own tools
3. Ensure commit messages follow conventions
4. Demonstrate real-world AI-assisted development

### How to Proceed?

After writing code, human developers can:

1. Use dscli to analyze code changes
2. Let dscli help write commit messages
3. Review and refine the commit message
4. **Indicate code source in the commit message**
5. Execute the commit

### Commit Message Format

Use conventional commit format, **must include code source**:

- `feat:` New feature
- `fix:` Bug fix
- `docs:` Documentation updates
- `style:` Code formatting adjustments
- `refactor:` Code refactoring
- `test:` Test-related
- `chore:` Build process or auxiliary tools

**Examples**:

```
feat: add batch file processing

Code source: Human-written core logic, Dscli-assisted testing

Implement batch file processing for the file tool,
supporting glob patterns and recursive directory traversal.
```

```
fix: resolve nil pointer panic in chat session

Code source: Dscli-assisted

Fix a nil pointer dereference when the session store
is not properly initialized.
```

3. Configure development environment

### 2. Create a Branch

- **Write code and tests (only human or Dscli-generated code accepted)**

### 4. Commit Changes

- Stage changes: `git add .`
- **Write commit messages (must use dscli assistance)**

### 5. Create a Pull Request

- Create a pull request
- **Describe code source in the PR description**

## 📋 Code Source Detail Standards

### 1. Acceptable Code Sources

#### 1.1 Human-Written Code

- **Definition**: Code directly conceived, designed, and written by a human developer
- **Requirements**:
  - Must be personally written by the contributor
  - Can reference existing code for design ideas, but cannot directly copy
  - Must understand the meaning and purpose of each line

#### 1.2 Dscli-Generated Code

- **Definition**: Code generated with the assistance of the dscli tool
- **Requirements**:
  - Must use dscli's chat or reasoner functionality
  - Must be reviewed and modified by a human after generation
  - Must ensure code conforms to project standards
  - Must indicate Dscli usage in the commit message

### 2. Unacceptable Code Sources

#### 2.1 Other AI Tools

- Code generated by ChatGPT, Copilot, Claude, etc.
- Any AI assistance tools other than Dscli

#### 2.2 Third-Party Code

- Code directly copied from other projects
- Unmodified open-source code snippets
- Code snippets from Stack Overflow, etc.

#### 2.3 Auto-Generated Tools

- Code generated by code generators
- Template engine auto-generated code
- Automation tool output without human review

### 3. Mixed Source Handling

#### 3.1 Human + Dscli Mixed

- **Allowed**: Human writes part of the code, Dscli assists with other parts
- **Requirement**: Must detail the source of each part in the commit message

#### 3.2 Dscli-Generated + Human-Edited

- **Allowed**: Dscli generates code, then human modifies and optimizes
- **Requirement**: Modifications must be significant, not just minor adjustments

### 4. Verification Methods

#### 4.1 Commit Message Verification

- Must include code source description
- Must specify which tools were used
- Must describe the development process

#### 4.2 Code Style Verification

- Check if code follows project standards
- Check for traces of other tools
- Check code logic consistency

#### 4.3 Process Record Verification

- Maintaining development process records is encouraged
- Dscli interaction history can be provided
- Helps reviewers understand the development process

## Testing Standards

### 1. Test File Structure

Test files must correspond one-to-one with source files:

```go
// Source file: tools.go
// Test file: tools_test.go
```

### 2. Test Function Naming

Use descriptive test function names:

```go
func TestFunctionName(t *testing.T) {
    // Basic functionality test
}

func TestFunctionName_EdgeCase(t *testing.T) {
    // Edge case test
}

func TestFunctionName_ErrorHandling(t *testing.T) {
    // Error handling test
}
```

### 3. Test Case Organization

Use table-driven tests:

```go
func TestFunctionName(t *testing.T) {
    testCases := []struct {
        name     string
        input    string
        expected string
        wantErr  bool
    }{
        // Normal path
        {"normal case", "input", "expected", false},
        // Edge conditions
        {"empty string", "", "", false},
        // Error path
        {"invalid input", "invalid", "", true},
    }

    for _, tc := range testCases {
        t.Run(tc.name, func(t *testing.T) {
            // Test logic
        })
    }
}
```

### 4. Test Coverage Requirements

- Overall coverage >= 80%
- Key functionality coverage >= 90%
- Error handling paths must be covered

### 5. Test Quality Metrics

| Metric | Requirement | Check Method |
|--------|------------|--------------|
| **Test Coverage** | >= 80% | `go test -cover` |
| **Edge Case Tests** | Must include | Code review |
| **Error Tests** | Must include | Code review |
| **Test Readability** | Clear and understandable | Code review |
| **Test Independence** | No external dependencies | Run in isolation |

## Release Process

### Version Number Rules

- Major version: Breaking changes, may not be backward compatible
- Minor version: New features, backward compatible
- Patch version: Bug fixes, backward compatible

### Release Steps

1. Ensure all tests pass
2. Create a Git tag (e.g., v0.4)
3. Tag message includes version highlights
4. Do not create separate release notes
5. Version information is tracked via Git tags and commit history

### Documentation Conventions

- Do not create unnecessary RELEASE_*.md files
- Version changes are tracked through commit history
- Keep the codebase clean, reduce maintenance burden
