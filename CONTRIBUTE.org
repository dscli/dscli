* 贡献者公约

欢迎为 dscli 项目做出贡献！本公约将帮助您了解如何参与项目开发，并确保项目保持高质量和一致性。

* 项目理念

dscli 是一个命令行工具，为 DeepSeek AI 提供本地化集成。我们的目标是：

1. 提供简洁、高效的 AI 辅助编程体验
2. 保持代码质量和可维护性
3. 实践我们推广的工具和方法论
4. 遵循最小化原则，避免不必要的复杂性

* 开发规范

** 代码风格
- 使用 Go 语言标准编码风格（gofmt）
- 遵循 Go 语言最佳实践
- 函数和变量使用有意义的命名
- 添加适当的注释和文档

** 文档规范
- 遵循最小文档原则，只维护必要的文档
- 避免创建不必要的发布说明文档
- 版本信息通过 Git 标签和提交历史获取
- 代码即文档，保持代码清晰可读

** 提交规范
- 提交信息使用约定式提交格式
- 每次提交应聚焦于单一功能或修复
- 鼓励使用 dscli 自身协助编写提交信息

* 提交规范

** 核心原则
我们鼓励使用 dscli 来协助编写提交信息，确保提交信息的规范性和一致性。

** 为什么这样做？
1. 保持项目一致性
2. 实践我们推广的工具
3. 确保提交信息的规范性
4. 展示 AI 辅助开发的实际应用

** 如何操作？
人类开发者编写代码后，可以：
1. 使用 dscli 分析代码变更
2. 让 dscli 协助编写提交信息
3. 人工审核并完善提交信息
4. 执行提交

** 提交信息格式
使用约定式提交格式：
- =feat:= 新功能
- =fix:= 错误修复
- =docs:= 文档更新
- =style:= 代码格式调整
- =refactor:= 代码重构
- =test:= 测试相关
- =chore:= 构建过程或辅助工具

示例：
#+begin_example
feat: 添加 deepseek-reasoner 模型支持

- 新增 --model 参数支持模型选择
- 添加 model_id 字段到数据库表
- 实现模型历史消息隔离
- 更新相关文档和示例
#+end_example

* 开发流程

** 1. 环境设置
1. 克隆仓库：=git clone https://gitcode.com/dscli/dscli.git=
2. 安装 Go 开发环境
3. 配置开发环境

** 2. 创建分支
- 从 =main= 分支创建特性分支
- 分支命名：=feat/xxx=, =fix/xxx=, =docs/xxx= 等

** 3. 开发实现
- 编写代码和测试
- 确保代码质量
- 可以使用 dscli 协助代码编写和审查
- 遵循最小文档原则

** 4. 提交更改
- 暂存更改：=git add .=
- 编写提交信息（鼓励使用 dscli 协助）
- 执行提交：=git commit -m "提交信息"=

** 5. 创建合并请求
- 推送到远程仓库
- 创建合并请求
- 等待代码审查

* 文档要求

** 最小文档原则
1. 只维护必要的文档，避免增加维护负担
2. 不创建不必要的发布说明文档
3. 版本信息通过 Git 标签和提交历史获取
4. 代码即文档，保持代码清晰可读

** 必须包含的文档
1. 代码注释和文档字符串
2. 用户使用文档（README.org）
3. API 文档（如有必要）
4. 贡献者公约（本文件）

** 文档格式
- 使用 Org mode 格式
- 源代码块使用正确的高亮
- 保持一致的标题层级
- 链接使用 Org mode 语法

* 测试要求

** 单元测试
- 关键功能应有单元测试
- 测试覆盖主要逻辑路径
- 测试文件使用 =_test.go= 扩展名

** 集成测试
- 测试命令行接口
- 测试与数据库的集成
- 确保向后兼容性

* 代码审查

** 审查要点
1. 代码正确性和安全性
2. 是否符合项目规范
3. 是否有适当的测试
4. 是否遵循最小文档原则
5. 提交信息是否规范

** 审查流程
1. 至少需要一名核心贡献者审查
2. 所有审查意见必须被处理
3. 通过审查后由维护者合并

* 发布流程

** 版本号规则
- 主版本号：重大变更，可能不向后兼容
- 次版本号：新功能，向后兼容
- 修订号：错误修复，向后兼容

** 发布步骤
1. 确保所有测试通过
2. 创建 Git 标签（如 v0.4）
3. 标签信息包含版本亮点
4. 不创建单独的发布说明文档
5. 版本信息通过 Git 标签和提交历史获取

** 文档约定
- 不创建不必要的 RELEASE_*.md 文件
- 版本变更通过提交历史追溯
- 保持代码库简洁，减少维护负担

* 沟通渠道

** 问题报告
- 使用 GitCode 的问题跟踪系统
- 提供详细的重现步骤
- 包括环境信息和相关日志

** 讨论交流
- 使用 GitCode 的讨论功能
- 保持专业和友好的交流氛围
- 尊重所有贡献者

* 行为准则

** 基本原则
1. 尊重他人
2. 建设性反馈
3. 包容多样性
4. 专业行为

** 不可接受行为
1. 人身攻击或侮辱性语言
2. 骚扰或歧视行为
3. 发布不当内容
4. 其他不专业行为
