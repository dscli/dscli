#+title: LLM Editor Design
#+date: <2026-03-06 五>

* 要解决的问题
  如何设计一套适合LLM“思维模式”的文件操作工具。已有的工具
  - write_file
  - read_file
  - search_file
  - read_file_with_line_range
  - write_file_with_line_range
  - search_file_with_pattern

  已经覆盖了基本操作，但LLM在使用它们时容易出现行号漂移、语法错误（如
  括号不匹配）等问题。这并非LLM能力不足，而是工具与LLM的“认知方式”不
  够匹配。本文先从LLM的特点出发，分析现有工具的局限性，再探讨更适合的
  工具设计。
* LLM的“认知特点”与现有工具的冲突
  - LLM擅长文本模式匹配与生成，但缺乏对“行号”等绝对坐标的精确记忆。

    人类使用行号是因为我们能直观理解“第10行”的物理位置，但LLM在对话
    中看到的文件内容会不断变化，行号随修改而变动，它难以维护一个准确的
    “行号地图”。工具中的 write_file_with_line_range 和
    read_file_with_line_range 都依赖行号，这会导致LLM在多次修改后出错。

  - LLM倾向于一次性输出完整修改，而非逐步“编辑”

    人类用Vim或VSCode时，会逐行移动光标、插入删除。但LLM更适合“整体思
    考后给出修改方案”，因为它是在文本空间里进行推理。如果工具要求它分
    步执行（比如先插入一行，再删除另一行），不仅增加了调用次数，还容易
    在步骤间丢失上下文。

  - LLM需要明确的“修改锚点”

    当LLM想“在函数foo的末尾添加一行print('done')”时，它脑中想的是
    “函数末尾”这个语义位置，而非一个具体的行号。现有工具只提供行号定
    位，迫使LLM去猜测行号，而它并不擅长此事。

  - LLM容易在局部修改中破坏语法结构

    你提到的括号不匹配，正是因为LLM在替换或插入代码时，未能正确感知周
    围代码的语法边界。如果工具不能帮助它保持语法完整性，错误就难以避免。
* 设计适合LLM的工具的原则
  基于以上分析，可以提炼出几条设计原则：

  - 基于内容而非坐标

    用文本块、正则表达式、语义标签（如函数名）作为操作锚点，避免使用绝
    对行号。

  - 支持“声明式修改”

    让LLM描述“我想要把A改成B”，而不是“删除第X行，然后在第Y行插入”。
    工具负责执行匹配和替换。

  - 提供结构感知能力

    帮助LLM了解代码的抽象语法结构（如函数、类的作用域），以便它能在正
    确的上下文中进行修改。

  - 具备语法校验与反馈

    修改后立即检查语法正确性，如果出错，将错误信息返回给LLM，让它有机
    会修正。

  - 支持回滚与试探

    允许LLM尝试多种修改方案，在失败时撤销。
* 工具集设计方案
  下面是一组可能更适合LLM的文件操作工具，可以根据实际需要裁剪和实现。

** 核心读取工具
  - read_file(file_path)
  读取整个文件内容（现有工具可保留）。

  - grep_file(file_path, pattern)
  返回匹配模式的行及行号（用于定位，但行号仅供LLM参考，不作为后续修改的绝对坐标）。

  - get_file_structure(file_path, language)
  利用语法分析（如tree-sitter）返回文件的大纲：函数名、类名、行号范围
  等。LLM可以据此决定在哪个函数内修改。

** 核心修改工具（避免行号）
  - replace_text(file_path, old_text, new_text, occurrence='all'/'first'/'last')
  在文件中查找 old_text（可以是精确字符串或正则表达式），替换为 new_text。可指定替换第几次出现。
  优点：基于内容匹配，行号变动不影响。

  - insert_text(file_path, anchor_text, new_text, position='after'/'before', occurrence='first')
  在第一个（或指定次）出现的 anchor_text 之前或之后插入 new_text。如果 anchor_text 是唯一标识（如函数签名），则非常可靠。

  - replace_block(file_path, start_pattern, end_pattern, new_block)
  查找从匹配 start_pattern 的行开始，到匹配 end_pattern 的行结束的代码块，整体替换为 new_block。这常用于替换整个函数或循环体。

  - write_file(file_path, new_content)
  全量覆盖（谨慎使用，但有时需要，此工具已有）。

** 高级工具（利用结构化信息）
  - apply_diff(file_path, unified_diff)
  接受标准unified diff格式的文本，应用修改。LLM可以直接输出diff，工具负责合并。这是许多版本控制系统和协作工具采用的方式，对行号变化有较好鲁棒性。

  - edit_function(file_path, function_name, new_body)
  通过语法树找到指定函数，将其函数体替换为 new_body（保留函数签名）。这需要编程语言支持，但能从根本上避免语法错误。

  - insert_import(file_path, import_statement)
  智能地在文件合适位置（如已有import块后）插入新的导入语句，避免重复。
** 辅助工具（提高成功率）
  validate_syntax(file_path, language)
  调用编译器/解释器检查语法，返回错误信息。LLM可在修改后调用此工具自我纠错。

  undo_last_change(file_path)
  回滚最后一次修改，用于LLM发现错误时恢复。

  get_file_hash(file_path)
  获取文件内容的哈希值，用于确认文件未被外部修改。

* 如何让LLM高效使用这些工具
  仅仅提供工具还不够，还需要设计合理的“使用流程”和提示词，引导LLM按
  步骤操作。例如：

  - 读取并理解当前代码
  调用 read_file 获取全貌（如果文件不大）。
  调用 get_file_structure 获取大纲，锁定要修改的函数。

  - 制定修改计划
  如果修改范围明确，优先使用基于内容的工具（replace_text、insert_text）。
  如果修改复杂，考虑生成diff或全量替换。

  - 执行修改并验证
  调用修改工具。
  立即调用 validate_syntax 检查，若有错误，让LLM分析并重新修改。

  - 迭代
  允许LLM多次修改，并利用撤销功能回退错误步骤。

  你可以在提示词中明确告诉LLM：“请避免使用行号，尽量使用文本匹配或函
  数名作为锚点。” 并示范工具的正确用法。
